### A new LLM feature/agent rebuilds platform LLM plumbing instead of using the sanctioned mechanism

- **Shape** — When reviewing a PR that adds or changes an LLM-powered feature/agent in a repo that already has platform conventions for LLM plumbing, run an explicit conformance pass on each plumbing concern *before* accepting a bespoke re-implementation — code that is internally correct and self-consistent can still silently establish a divergent third convention and open a governance gap. Concerns to check: (a) **prompts** — a system/tool prompt hard-coded as a literal string in code vs the platform's hosted/versioned prompt mechanism (Braintrust prompt management / a prompt-registry package), which carries versioning + eval-override governance the literal string skips; (b) **tool/function definitions** — hand-rolled provider-format dicts + a handler-dispatch dict + direct vendor SDK calls vs the sanctioned tool framework siblings use (fastmcp/MCP decorators with type-inferred schemas, or LangChain `@tool`/`bind_tools`); (c) **cross-cutting primitives** — PII redaction / content-safety regexes re-implemented locally vs a shared core module that already owns them (and may already be duplicated elsewhere, signalling a known consolidation intent). Also check for **term collisions**: a module/package name ("guardrails", "session") that already means something *different* in the shared package (e.g. "guardrails" = LLM jailbreak prompt-hardening, not PII/permission gating) is a documentation hazard worth flagging.
- **Trigger signals** — a new/changed LLM feature or agent; a system/tool prompt present as a literal Python string with no slug/version; tool definitions written as hand-authored provider dicts plus a `TOOL_HANDLERS`-style dispatch and direct `anthropic`/`openai` (or other vendor) SDK calls; a local `pii`/`redact`/`guardrail`/`content`-safety regex module; the consumer's deps pull only `[observability]`/`[database]`-style extras, not the `[llm]`/`[prompts]` extras the sibling LLM services use; module or schema names that overload an established shared concept. Sibling services (in this repo: astro, ai_mod) use `fastmcp-slim`/LangChain tools, `dscout_core_py.llm.BraintrustPrompt`, and `dscout_core_py.observability.pii_masking` — diverging from all three at once is the smell.
- **Evidence**
  - `- {type: missed, ref: PR #25914, date: 2026-06-02}` — re-review of a new support-agent service approved (verdict APPROVE) on correctness/security grounds; the user surfaced four structural misses the review never raised: system prompt hard-coded in `system_prompt.py` instead of Braintrust prompt management (used pervasively in astro; service didn't even pull `dscout-core-py[llm]`); tool definitions hand-rolled as `TOOL_DEFINITIONS` dicts + `TOOL_HANDLERS` dispatch + direct Anthropic/OpenAI SDK calls instead of fastmcp/LangChain (the repo's two sanctioned tool patterns); local `guardrails/content.py` PII regexes as a third copy of `dscout_core_py.observability.pii_masking`; and "guardrails" colliding with the shared package's existing (different) `llm/guardrails` concept.
- **Proposed promotion** — `target: references/general-checklist.md` (a cross-cutting "LLM-plumbing conformance" review category); `wording:` "For a PR that adds/changes an LLM feature or agent, run a conformance pass against the platform's sanctioned LLM plumbing before accepting bespoke code: prompts via the hosted/versioned prompt registry (not literal strings); tool/function definitions via the established tool framework (fastmcp/MCP or LangChain), not hand-rolled provider dicts + direct SDK calls; PII/content-safety via the shared core primitive, not a re-implemented regex copy. Flag module-name collisions with shared concepts. Correct, self-consistent LLM code can still establish a divergent convention and skip prompt/eval governance."
- **Status** — pending

### A flag-gated / lazily-loaded data source feeding a non-gated consumer fails open

- **Shape** — A UI affordance that restricts or alters behavior (lock, gate, hide, disable, redirect, enforce) is derived from data loaded only by a *conditional* fetch — feature-flagged, lazily-triggered, or gated on some other precondition. When the consumer UI is reachable on a path that is NOT behind the same condition, the absent/unresolved data silently collapses to the affordance's falsy/default branch, so the restriction just doesn't apply. The dangerous case is when "data absent" coincides with the *permissive* outcome (e.g. `null` → unlocked / visible / allowed) and there is no fail-closed fallback and no "still loading vs. genuinely empty" distinction.
- **Trigger signals** — a lock/gate/hide/disable/enforce decision reads a value resolved from a store/cache/context that is populated only by a flag-gated or lazily-triggered fetch; the consumer component (modal, page, route) is reachable on a path NOT behind the same flag/precondition; the "absent" branch and the "permissive" branch are the same code path; condition skew is plausible (flag toggled off mid-session, staged rollout, targeting change, race before load); no explicit fail-closed default or unresolved-vs-empty distinction.
- **Evidence**
  - `- {type: caught, ref: PR #26006, date: 2026-06-02}` — an export-modal field lock was derived from a store populated only by a feature-flagged fetch, while the modal itself wasn't flag-gated; with the flag off but a value already applied, the lookup resolved to null and the lock silently failed open (field editable instead of locked). Bounded in practice but a real fail-open edge surfaced only by tracing the sole fetch caller's gating.
- **Proposed promotion** — `target: references/general-checklist.md` (data-integrity / correctness check); `wording:` "A restrictive UI affordance (lock/gate/hide/disable) derived from conditionally-loaded data (feature-flagged or lazy fetch) must fail *closed* or explicitly distinguish 'unresolved' from 'empty' — if the consumer UI is reachable without the same gating, absent data collapses to the permissive branch and the restriction silently doesn't apply. Trace the sole loader's gating against every path that reaches the consumer."
- **Status** — pending

### Acceptance-criterion behavior ships in untested glue while tests cluster on an adjacent layer

- **Shape** — The literal ticket/spec acceptance criterion is realized in wiring/glue code (prop threading, conditional render, an event handler, a small mapping), but the PR's new/changed tests concentrate on an adjacent, easier-to-test layer (a store, reducer, pure mapper, helper). Because the adjacent tests are thorough and non-vacuous, they create an impression of coverage that masks the fact that the criterion *itself* — the end-to-end behavior the ticket asked for — has zero direct assertion.
- **Trigger signals** — the PR links a ticket/spec with a concrete acceptance criterion; new or modified tests are concentrated on a sub-layer (store/reducer/util/selector) rather than the component/endpoint/handler that realizes the criterion; the criterion depends on glue no test exercises end-to-end; reviewers (or a spec-reviewer bot) mark the requirement "met" from reading the code rather than from a test that asserts it.
- **Evidence**
  - `- {type: caught, ref: PR #26006, date: 2026-06-02}` — store-layer slot-resolution tests were comprehensive, but nothing asserted the actual acceptance criterion (a form field locks + pre-populates from the resolved record); that behavior lived entirely in untested modal glue.
- **Proposed promotion** — `target: references/general-checklist.md` (QA / requirements-coverage check); `wording:` "When a PR links a ticket with a concrete acceptance criterion, check that a test asserts the criterion's *end-to-end* behavior — not just an adjacent sub-layer (store/util/mapper). Thorough sub-layer tests can mask that the criterion itself ships in untested glue (prop threading, conditional render, handler). 'Code looks right' ≠ 'criterion is asserted'."

### Cross-service mapper bug masked by a test fixture richer than the real client's wire shape
- **Shape** — A service-A → service-B integration (e.g. Astro MCP tool → Axon endpoint) carries a payload built by a mapper on side A. Side B's mapper/parser reads fields in a particular shape. Side B's unit tests hand-supply a *richer* shape than side A's mapper actually emits (extra keys, maps where the real client sends bare scalars, a populated field the real client omits), so the tests pass green while the live cross-service path silently drops/corrupts data. The bug is invisible because the test fixture and the real client disagree on shape, and only the fixture is exercised.
- **Trigger signals** — a PR adds/changes an endpoint fed by another service's serializer/mapper; the consumer's tests construct input by hand rather than from the producer's actual output; the producer's schema/`_build_*payload*` emits `list[str]`/omits a field/uses a flat shape, while the consumer's mapper does `Map.get(c, "label")`/expects the field/expects a nested shape; "the field round-trips" is asserted only at the producer (payload-emission) layer, never end-to-end through the consumer's HTTP/translation path.
- **Evidence**
  - `- {type: caught, ref: MCP-265 session, date: 2026-06-10}` — `_build_payload_from_plan` omitted question `position`; Axon defaulted all to 0 → order lost + `child_blocks` misclassified as top-level questions. Axon tests hand-supplied `position: 1/2`.
  - `- {type: caught, ref: MCP-265 session, date: 2026-06-10}` — `ProposedScreenerQuestion.choices: list[str]` but Axon `map_single_screener_question` does `Map.get(c, "label")` → nil labels. Axon tests hand-supplied `%{"label" => ...}` choice maps.
- **Proposed promotion** — `target: references/cross-service-contracts.md`; `wording:` "For an endpoint fed by another service's serializer/mapper, verify the consumer's expected field shapes against the PRODUCER's actual emitted shape (read its `_build_*payload*`/serializer literally) — NOT against the consumer's test fixtures, which often carry a richer shape than the real client sends. Require at least one integration test that drives the real client's wire shape through the full HTTP/translation path, not the context function with hand-built inputs."

### Event-emit path is executed by existing tests but the fire/silent/payload decision is never asserted

- **Shape** — A PR adds an event/message emit — often conditional (gated on a pre-vs-post diff, a threshold, a dedupe, a state change). Existing tests already drive the surrounding code path and the emit line *runs* (commonly because the dispatch helper is injectable and isn't mocked), so coverage tools report it covered. But the assertions target resulting state or downstream side effects only — never *whether* the event fired, *how many times*, or *with what payload*. The actual emit decision (fire-once / silent-on-no-change / correct ids) is therefore unverified despite apparent coverage.
- **Trigger signals** — a PR adds a conditional/gated emit (diff-gated, threshold-gated, dedupe-gated, fire-on-change); the emit sits inside a code path existing tests already exercise; the dispatch/emit helper is injectable/unmocked so it executes during tests; existing assertions check state or side effects, not emitted events; no assertion on fire-count, no-fire-on-no-change, or payload contents.
- **Evidence**
  - `- {type: caught, ref: PR #26006, date: 2026-06-02}` — a new diff-gated event emit ran inside tests that already drove the node, but those tests asserted only on resulting state; the fire-once-on-change / silent-on-no-change behavior — the load-bearing logic — had no assertion.
- **Proposed promotion** — `target: references/general-checklist.md` (QA / test-fidelity check); `wording:` "For a newly-added conditional emit (diff/threshold/dedupe-gated event or message), check that a test asserts the *emit decision* — fired vs. not, fire-count, and payload — not just resulting state. Existing tests often execute the emit line (unmocked dispatcher) and report it 'covered' while never verifying whether it fires correctly."
- **Status** — pending

### A "mirror"/"ported" constant or formula must be verified against the CURRENT upstream, not the PR's base

- **Shape** — When a change ports or mirrors a formula, schema, set of constants, or any value from another file/service into this one (especially across a language boundary), verify it against that source **as it exists on the remote default branch right now** — not the copy present at the PR's base. The mirrored source is usually not in the diff, so a reviewer naturally diffs against the branch's own snapshot and confirms parity that was true at branch time but has since gone stale. Pull the upstream file at the default-branch ref and check its recent commit history for re-tunes the branch predates; a mirror can be exactly correct when written and silently wrong by merge time.
- **Trigger signals** — the PR description says it "mirrors"/"matches"/"ports"/"keeps in sync with" another file; the mirrored source is NOT in the diff; the PR's stated purpose is parity between two sides; the upstream source has commits dated after the PR branch's base; another reviewer reports "verified term-by-term against \<file\>" (they likely checked the base snapshot, not the current upstream).
- **Evidence**
  - `- {type: caught, ref: PR #25998, date: 2026-06-02}` — backend ported a frontend estimate formula and matched it exactly *at the branch base*; a prior reviewer confirmed term-by-term parity. The upstream file had been re-tuned on the default branch after the base (a separate merged PR), so merging would have re-created the very cross-side mismatch the change existed to eliminate (~2–3.6× off, plus a dropped additive term). Caught only by fetching the upstream file at the default-branch ref and reading its commit history.
- **Proposed promotion** — `target: references/cross-service-contracts.md`; `wording:` "When a change mirrors/ports a value (formula, schema, constants) from another file or service, verify it against that source at the **current remote default-branch ref**, not the copy at the PR's base — and scan the source's recent commit history for changes the branch predates. Parity confirmed against the branch snapshot (even by another reviewer) does not prove parity at merge time; an un-rebased mirror of a since-changed source ships stale."
- **Status** — pending

### Entity-level flag derived per-sublist when the same function runs on partitions of one set

- **Shape** — When a function infers an entity-level / scope-level boolean from the contents of the collection it is handed (e.g. `any(items, predicate)`), and that same function is invoked separately on partitions of one logical set (complete vs. incomplete, page slices, filtered subsets, before/after a cursor), the flag is re-derived per call and can flip between partitions — yielding inconsistent treatment of items that belong to the same entity. The correct flag is a property of the whole set and must be computed once over the full collection and threaded into each partitioned call.
- **Trigger signals** — a function takes a collection and computes a flag via `any()`/`some()`/`exists?` over it; the same function is called more than once on split/filtered/paginated subsets of a single logical set; the marker elements that satisfy the predicate can land entirely in one partition (e.g. all sorted to the front, or consumed as a session progresses); the function already accepts (or could accept) an override param for the flag.
- **Evidence**
  - `- {type: caught, ref: PR #25998, date: 2026-06-02}` — a progress calculation called an effort function separately on "complete" and "incomplete" item lists; the function derived an "is this an X-type entity" flag from whether *that list* contained a marker item. Once the marker items were all consumed, the incomplete list derived the flag as false and mis-priced the remainder, skewing the ratio mid-session. Fix: compute the flag once over the full set and pass it via the override the function already exposed.
- **Proposed promotion** — `target: references/general-checklist.md` (correctness check); `wording:` "If a function derives an entity/scope-level flag from the contents of the collection passed to it (`any(...)`), and the same function is called on partitions of one logical set (complete/incomplete, pages, filtered subsets), the flag can flip per partition. Compute it once over the full set and thread it into each call."
- **Status** — pending

### Verify library-API claims against the LOCKED version's source, not current docs

- **Shape** — Before asserting (or DROPping) a finding that a code path calls a non-existent / wrong / renamed library method, verify the method against the **source of the version actually locked** (`uv.lock` / `package-lock` / `Gemfile.lock`), not against current online docs or memory. Major-version renames and package restructures make docs and intuition stale — the locked version is what runs. A method that "doesn't exist" in the docs may be the canonical one at the pinned version (or vice-versa).
- **Trigger signals** — a review finding hinges on a library method name/signature existing or not; the claim is grounded in web docs, a different version's source, or memory rather than the pinned version; the dependency has had a recent major version bump or known API rename; the symptom would be an AttributeError/NoMethodError at runtime that no test covers.
- **Evidence**
  - `- {type: caught, ref: PR #25913, date: 2026-06-01}` — review drafted a blocking "calls non-existent `get_embedding_dimension()`; correct API is `get_sentence_embedding_dimension()`" finding from sbert.net docs. Checking `uv.lock` (sentence-transformers 5.5.1) showed the package was restructured: `get_embedding_dimension()` is now canonical and `get_sentence_embedding_dimension()` is the `@deprecated` alias. The locked-source check converted a false blocker into a correct DROP.
- **Proposed promotion** — `target: gotchas.md`; `wording:` "**Category:** failure-mode. **Context:** a finding (or a DROP of one) turns on whether a library method/signature exists. **Wrong:** asserting it from current online docs, a different version, or memory. **Right:** read the source of the version pinned in the lockfile (uv.lock / package-lock / Gemfile.lock) — major-version renames and restructures make docs stale; the locked version is what runs. **Why:** both false blockers and false DROPs come from version-skewed API knowledge; the lockfile is the only authoritative reference for what the code actually calls."
- **Status** — pending

### Unbounded prefix match/delete without a delimiter boundary over-matches sibling keys

- **Shape** — When a query matches or deletes rows by a string **prefix** (`col.startswith(x)`, `LIKE 'x%'`, `key LIKE prefix||'%'`), check that the prefix is anchored on a structural delimiter and that LIKE wildcards in the prefix are escaped. Without a boundary, a prefix that is a substring-prefix of a longer sibling key (variable-width IDs, path-like keys) matches/deletes the sibling too — silent over-deletion / data loss on delete paths, false positives on read paths.
- **Trigger signals** — a `delete`/`select`/`where` uses `startswith` / `LIKE 'prefix%'` / `ilike` against a structured key column; IDs follow a `a:b:c:chunk:n` or path-like scheme and are **not fixed-width**; the prefix is caller-supplied without a trailing delimiter; `startswith`/`like` is used without `autoescape=True` on keys that can contain `_` or `%`.
- **Evidence**
  - `- {type: caught, ref: PR #25913, date: 2026-06-01}` — `delete_document(doc_id_prefix)` used `Document.id.startswith(doc_id_prefix)` and was called with bare doc ids; because chunk IDs are `{doc_id}:chunk:{n}` and doc ids aren't fixed-width, purging `runbooks:api` would also delete `runbooks:api-v2:chunk:0`. Fix: anchor on `:chunk:` and `autoescape=True`.
- **Proposed promotion** — `target: references/general-checklist.md` (data-integrity check); `wording:` "Prefix matches/deletes (`startswith`, `LIKE 'x%'`) against structured/variable-width key columns must anchor on a structural delimiter (e.g. `f\"{prefix}:chunk:\"`) and escape LIKE wildcards (`autoescape=True`) — an unanchored prefix over-matches sibling keys (silent data loss on delete paths)."
- **Status** — pending

### New-service / scaffold PRs: check conformance to established sibling-service conventions, not just in-diff correctness

- **Shape** — When reviewing a PR that stands up a **new service or scaffold** in an established (esp. polyglot mono-)repo, run an explicit conformance pass against how *existing sibling services* are built and deployed — don't just verify the correctness of the code in the diff. The common miss is approving a self-consistent scaffold that silently diverges from house patterns (build/deploy/config/infra).
- **Trigger signals** — a PR adds a new app/service directory, its own `Dockerfile` / `docker-compose` / config module / settings class / observability or other `core/` shim; a docstring asserts an **operational/infra fact** ("its own dedicated DB instance", "isolated") with **no infra (Terraform/compose/manifest) in the diff** to back it; the new service is a consumer of shared prod infra (a shared DB, queue, cache). Conformance axes to check against siblings: shared orchestration / compose integration (+ shared dependency services, image/version pins); container build convention (multi-stage, shared/base image); config & env-loading convention (single source of truth); thin re-export "seams" that add no value vs how siblings import shared libs; and **prod data-store / infra provisioning + capacity** (does a new consumer land on already-loaded shared hardware?).
- **Evidence**
  - `- {type: missed, ref: PR #25912, date: 2026-05-29}` — reviewed a new-service scaffold multiple times, anchoring on in-diff correctness (a migration); missed five conformance gaps — divergence from sibling-service orchestration/compose, container-build, config, and observability conventions, plus a docstring asserting a *dedicated* data-store instance with no infra in the diff to back it (against shared prod hardware). The PR author surfaced all five, not the reviewer.
  - `- {type: missed, ref: PR #25914, date: 2026-06-02}` — same support-agent service, PR 3/4: review anchored on correctness/security (verifying prior fixes) and missed sibling-convention divergence across the LLM plumbing — hosted prompt registry, tool-definition framework, and shared PII primitive (detailed in the separate LLM-plumbing-conformance entry). Reinforces that new-service/scaffold PRs need a dedicated conformance pass against siblings — and that the conformance axes must include domain-specific platform conventions (here, LLM plumbing), not just orchestration/build/config/infra.
- **Proposed promotion** — `target: references/general-checklist.md` (a cross-cutting "new-service conformance" review category); `wording:` "For a PR that introduces a new service/scaffold, run a conformance pass against existing sibling services — orchestration/compose integration (+ shared deps, image/version pins), Dockerfile multi-stage/base convention, config & env-loading convention, no-value re-export shims, and prod DB/infra provisioning + capacity (a new consumer on shared/loaded hardware) — and challenge any docstring that asserts an operational/infra fact with no infra in the diff. Don't approve a self-consistent scaffold that diverges from house patterns."
- **Status** — pending

### Verify execution order / privilege context before recommending a fix location in a shared runner

- **Shape** — When recommending a fix whose correctness depends on *when* or *with what privileges* a shared runner/framework invokes a hook, trace the actual call order (and transaction/privilege context) before asserting the fix location. A fix placed in a leaf hook (e.g. a migration's `upgrade()`) can be too late if the framework runs other schema-qualified or privileged bookkeeping first.
- **Trigger signals** — a review suggests adding bootstrap/setup code (schema/namespace creation, extension/plugin install, table or registry init, env/lifecycle setup) to a leaf hook owned by a shared runner (a migration framework like Alembic, a job/queue runner, an app lifecycle/startup sequence); a managed-resource privilege model or transaction/savepoint wrapping is in play; the fix's effectiveness hinges on the framework's invocation order rather than the leaf code itself.
- **Evidence**
  - `- {type: missed, ref: PR #25912, date: 2026-05-29}` — review recommended adding `CREATE SCHEMA` to a migration's `upgrade()`; re-review found the shared migration runner creates schema-qualified bookkeeping (a version-tracking table) *before* `upgrade()` runs, so the leaf fix was a no-op for fresh deploys.
- **Proposed promotion** — `target: references/general-checklist.md` (a cross-cutting "fix-suggestion" review check); `wording:` "When a review's suggested fix depends on execution order or privileges in a shared runner/framework the author didn't write, trace the actual invocation order (and transaction/privilege context) before asserting *where* the fix goes — a leaf-hook fix can run too late."
- **Status** — pending
